# C3: FORM-Style Sliding Window Smoothing for Odometry

> IV-GICP의 세 번째 기여 — 실시간 odometry 전용 윈도우 스무딩 전략

---

## 1. 배경: FORM이란?

**FORM** (Fixed-lag Odometry with Reparative Mapping, Potokar et al. 2025, arXiv 2510.09966)

핵심 메커니즘:
1. 매 프레임 ICP 후, 최종 GN iteration의 Hessian H를 저장
2. H의 eigenvalue = 각 DOF의 **observability** (큰 eigenvalue = 잘 관측됨)
3. 슬라이딩 윈도우 K프레임을 **factor graph**로 jointly optimize
   - 각 odometry factor의 information matrix = ICP Hessian
   - 퇴화 DOF(낮은 eigenvalue)를 인접 프레임의 높은 eigenvalue로 보정
4. 보정된 포즈로 **맵도 소급 수정** (Reparative Mapping)

왜 빠른가:
```
GTSAM factor graph: Levenberg-Marquardt + 자료구조 관리 → 수 ms
직접 6K×6K GN 시스템 조립: 3 iterations → <0.5ms (K=10)
```

## 2. 현재 구현 상태

### 구현된 것
- `pipeline.py`: `_window_smooth(T_new, T_rel, H, frame_id)` — K프레임 윈도우 pose 최적화 (Step 1)
- `iv_gicp.py`: `info["hessian"] = H` — ICP Hessian 반환
- `factor_graph.py`: `add_odometry_factor(omega=H)` — Hessian을 information matrix로
- `flat_voxel_map.py`: `FlatVoxelMap.apply_delta_transform(R, t)` — O(V) 분포 보정 (구현됨)

### Step 2 (맵 소급 수정) — 구현됐으나 per-frame path에서 제거

**제거 이유 (2026-03-06 실험으로 확인):**
- 슬라이딩 윈도우 overlap: 프레임 k는 연속 window_size개의 윈도우에 모두 포함됨
- → 동일 프레임의 correction이 ~window_size번 맵에 적용 → 누적 오차 폭발
- SubT Final_UGV1: Step 2 활성화 시 ATE 0.065m → 0.321m (5배 악화)
- GEODE 터널: correction이 near-zero (일관된 퇴화 방향)라서 겉으로는 neutral이었음
- **결론: per-frame Step 2는 근본적으로 잘못된 설계. 루프 클로저 one-shot 이벤트에서만 안전.**

| Method | ATE |
|--------|-----|

| KISS-ICP | 9.112m |

→ Metro 터널에서 C2(intensity) + FORM 시너지로 KISS-ICP 대비 23% 개선!

### 왜 neutral인가?

**조건 1 (불충족): 퇴화 방향의 다양성**

FORM이 효과적이려면 윈도우 내 인접 프레임들이 **서로 다른 방향**에서 퇴화해야 함:
```
프레임 i: tunnel 방향 v₁에서 퇴화 (λ₁ ≈ 0)
프레임 j: 교차로에서 v₂ 방향에서 퇴화 (v₂ ≠ v₁)
→ 합산 information F_total·v₁ = H_i·v₁ + H_j·v₁ > 0 (j가 v₁ 복구)
```

현재 데이터셋의 문제:
- **KITTI 야외**: 모든 DOF가 이미 well-constrained → 추가 정보 없음
- **GEODE 직선 터널**: 500프레임 내내 **동일한 방향**으로 퇴화 → K=10 합산해도 null space 동일
- **SubT 지하**: KITTI와 유사하게 충분한 geometry

수학적으로:
```
F_total = Σ_k H_k
만약 ∀k: H_k · v ≈ 0  →  F_total · v ≈ 0
→ 최적화가 방향 v를 개선할 수 없음
```

**조건 2 (불충족): 맵 소급 수정**

포즈를 수정한 후 맵을 수정하지 않으면:
- 프레임 k의 포즈가 T_k → T_k' 로 보정됨
- 하지만 맵의 voxel들은 여전히 T_k 기준으로 빌드
- 다음 프레임 k+1의 registration이 T_k' 기준 소스 vs T_k 기준 타겟 → 불일치
- 이 불일치가 보정 효과를 상쇄

## 4. FORM이 실제로 도움 되는 시나리오 (odometry용)

### 4-1. 환경 전환 구간

```
[직선 터널] → [T자 교차로] → [옆 통로]
   프레임 i         프레임 j        프레임 k
   v₁ 퇴화         v₁, v₂ 관측     v₂ 퇴화
```

- 프레임 j가 양방향 관측 → 윈도우 내에서 i의 v₁, k의 v₂ 모두 복구 가능
- **지하철 환승역, 광산 갈림길, 건물 출입구** 등

### 4-2. 곡선 구간

```
[직진 터널] → [완만한 커브] → [직진 터널]
```

- 커브 구간에서 퇴화 방향이 점진적으로 회전
- K=10 윈도우 내에서 퇴화 방향이 30-45° 이상 변하면 complementary
- 커브 반경 R, 프레임 간격 d: 퇴화 방향 변화 ≈ arctan(K·d/R)

### 4-3. 비정형 구조물 통과

```
[평평한 벽] → [파이프/배관/환풍구] → [평평한 벽]
```

- 비정형 구조물 구간에서 기하학적 특징이 풍부 → 잠시 well-constrained
- 해당 프레임이 윈도우 내 다른 프레임들의 퇴화 보정에 기여

## 5. 최적 활용 전략 (코드 수정 없이)

### 전략 A: 조건부 활성화 (권장)

현재 코드: `IVGICPPipeline(window_size=10)`으로 항상 활성화

더 나은 접근:
```
1. 매 프레임 Hessian H의 condition number κ = λ_max/λ_min 모니터링
2. κ > threshold (예: 1000) → 현재 구간 퇴화 판정
3. 퇴화 진입/탈출 구간에서만 window smoothing 가중치 높임
4. 잘-constrained 구간에서는 window 효과가 neutral → overhead만 발생
```

이점: overhead 최소화 + 필요할 때만 작동

### 전략 B: 맵 소급 수정 연동

현재 누락된 Step 2 구현 시 기대 효과:
```
1. _window_smooth()로 K포즈 보정 → [T_1,...,T_K] → [T_1',...,T_K']
2. 각 프레임의 포즈 변화 ΔT_k = T_k'^{-1} @ T_k 계산
3. 해당 프레임의 voxel 기여분을 ΔT_k로 transform:
   μ_new = R_Δ · μ + t_Δ
   Σ_new = R_Δ · Σ · R_Δᵀ   (Theorem 2 — propagation error bound)
4. 수정된 voxel로 KDTree 재빌드
```

비용: O(V) per window (V = active voxels) — 전체 포인트 재삽입 O(N)보다 훨씬 저렴
이것이 원래 C3의 핵심 기여.

### 전략 C: 윈도우 크기 적응

고정 K=10 대신:
```
퇴화 심함 (κ > 5000): K=20 (더 많은 인접 프레임에서 정보 수집)
약간 퇴화 (κ 100~5000): K=10 (기본)
잘-constrained (κ < 100): K=1 (비활성화, overhead 제거)
```

## 7. 논문에서 C3 서술 방향

### 현재 문제
- C3를 "루프 클로저용"으로 소개 → odometry에서 검증 불가
- 리뷰어: "C3 실험이 없다" 지적 예상

### 제안: Odometry 전용 FORM-style C3로 재정의

```
기존 서술:
"C3: Loop closure 시 O(V) retroactive map update"

수정 서술:
"C3: Sliding-window odometry smoothing with retroactive map refinement
 — ICP Hessian을 Fisher information으로 활용하여
 — 퇴화 DOF를 인접 well-constrained 프레임에서 복구하고
 — 보정된 포즈로 맵 분포를 O(V)에 소급 수정"
```

이렇게 하면:
- 루프 클로저 불필요 (pure odometry)
- FORM과의 차이: 우리는 **intensity-augmented** Hessian → Theorem 1과 자연스럽게 연결
  - 기하 퇴화 시 intensity가 Hessian 보강 → 윈도우 내 information이 더 풍부
  - FORM은 geometry-only Hessian → 동일 퇴화 환경에서 정보 부족
- C1(adaptive voxel) + C2(intensity) + C3(window smooth + map update) = 통합 프레임워크

## 8. FORM vs IV-GICP C3 차이점 (논문용)

| | FORM | IV-GICP C3 |
|---|---|---|
| Information matrix | Geometry-only ICP Hessian | **4D geo-photometric Hessian** (C2) |
| Degeneracy handling | Hessian에 자연스럽게 인코딩 | **Theorem 1: intensity가 null space 보완** |
| Map update | Point-level reprocessing | **Distribution-level O(V) propagation** |
| Voxel resolution | Fixed | **Adaptive (C1)** |
| Backend | GTSAM factor graph | Lightweight 6K×6K direct GN |

핵심 논점: FORM은 geometry-only라서 **모든 프레임이 동일 방향 퇴화하면 무력**.
IV-GICP는 C2(intensity)가 Hessian을 보강하므로, 동일 환경에서도 **null space가 더 작음**.
→ 윈도우 스무딩의 유효 범위가 FORM보다 넓다.

---

## 9. 결론

**FORM 윈도우 스무딩은 "무조건" 좋은 게 아니라 "조건부"로 좋다:**

1. 인접 프레임의 퇴화 방향이 달라야 효과 있음
2. 맵 소급 수정(Step 2) 없이는 효과 제한적
3. IV-GICP의 강점: C2(intensity)가 Hessian 보강 → 순수 FORM보다 유효 시나리오 넓음

**Odometry용 활용 방안:**
- Step 2 (맵 소급 수정) 구현이 핵심 — 이것이 C3의 진짜 기여
- 조건부 활성화 (condition number 기반)로 overhead 최소화
- 논문에서는 "FORM + intensity + adaptive voxel" 통합으로 차별화
